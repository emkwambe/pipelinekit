# ADR-025 — Volume Anomaly Detection

**Date:** June 29, 2026  
**Status:** Accepted  
**Capability:** QM-6  
**Author:** Eddy Mkwambe + Command Center  
**Deciders:** Eddy Mkwambe

---

## Context

QM-4 (Coverage Monitoring) tells engineers how much of their pipeline
is tested. QM-6 answers the next question: "Is the data volume normal?"

A pipeline can pass all quality checks and still produce wrong results
if the volume is dramatically wrong — a source table that usually has
50,000 rows suddenly has 500, or 500,000. These volume anomalies are
silent failures that dbt tests and Soda checks do not catch by default.

Research finding (Q20, R-020): Enterprise pipelines fail 4.7 times per
month on average. Volume anomalies — data drops, duplicate loads,
missing partitions — are among the most common failure patterns.

QM-6 adds a baseline row count store and deviation detection. When a
pipeline run produces row counts significantly different from the
established baseline, PipelineKit flags it as an anomaly.

---

## Decision

Store row count snapshots in `state.db` in a new `qm_row_counts` table.

Two operations:

**Record:** After a pipeline run, record the row count for each table
into `qm_row_counts` with a timestamp. This builds the baseline over time.

**Check:** Compare the current row count against the rolling average of
the last N snapshots. If the deviation exceeds a configurable threshold
(default 20%), flag as an anomaly.

The CLI surface:

```bash
# Record current row counts (run after pipelinekit run)
pipelinekit quality record-counts --blueprint stripe-to-snowflake

# Check for anomalies against baseline
pipelinekit quality check-anomalies --blueprint stripe-to-snowflake

# Show row count history
pipelinekit quality row-count-history --blueprint stripe-to-snowflake
```

**Threshold:** Default 20% deviation triggers an anomaly warning.
Configurable per table via `--threshold` flag.

**Baseline window:** Default last 7 snapshots. Configurable via
`--window` flag. Minimum 3 snapshots required before anomaly detection
activates (not enough history before that).

**No AI calls** — deviation detection is purely statistical.
AI-assisted anomaly narrative is appropriate for QM-8 (Quality Scorecard),
not QM-6.

---

## Alternatives Considered

**Alternative 1: Use dbt source freshness**
dbt has a `source freshness` command that checks data age.
Rejected: freshness checks data recency, not volume.
Volume anomalies (wrong row count) are a different problem.

**Alternative 2: Connect to warehouse and count live**
Query Snowflake/DuckDB directly to get current row counts.
Rejected for QM-6: requires warehouse credentials and a live connection.
QM-6 must work offline with recorded data.
Live warehouse queries are appropriate for QM-7 (Schema Drift Detection).

**Alternative 3: Statistical outlier detection (z-score)**
Use z-score or IQR-based outlier detection.
Rejected: requires more than 7 samples to be meaningful.
Simple percentage deviation is more interpretable for engineers
and requires fewer samples to activate.

---

## Consequences

**Positive:**
- Catches silent volume failures that quality checks miss
- Builds a historical record of pipeline output volumes
- Enables QM-8 (Quality Scorecard) — needs volume trend data
- Directly addresses R-020 finding: most common pipeline failure patterns

**Negative:**
- New state.db table — non-destructive migration
- Requires manual `record-counts` step after each run
  (until pipeline run integration is built in a later sprint)
- First 3 runs produce no anomaly detection (insufficient baseline)

**Neutral:**
- No external dependencies — fully offline
- Does not require warehouse connection
