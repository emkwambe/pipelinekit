# Cross-Database Compatibility

Blueprint #003 builds against multiple warehouses without forking SQL: Snowflake (the `prod`/`dev` dbt targets) and DuckDB (the `local` target used by `scripts/verify-blueprint-003.ps1 -Local`), with BigQuery handled too. The trick is to keep all adapter-specific SQL behind dbt macros in `transform/macros/cross_db.sql` and call those macros from the staging models instead of writing warehouse-specific functions inline. Each macro dispatches on `target.type` at compile time — for example `{{ to_timestamp('created') }}` compiles to `to_timestamp_ntz(created)` on Snowflake, `to_timestamp(created)` on DuckDB, and `TIMESTAMP_SECONDS(created)` on BigQuery, while `{{ safe_boolean('refunded') }}` resolves the `'true'`/`'false'` string-vs-native-boolean difference between DuckDB and Snowflake. This means a model like `stg_stripe__charges.sql` is written once and verifies locally against synthetic DuckDB data yet still runs unchanged in production Snowflake. Future blueprint authors should follow the same pattern: never call a warehouse-specific function (epoch conversion, boolean casting, date truncation, JSON extraction, etc.) directly in a model — add or reuse a `target.type`-dispatching macro in `cross_db.sql` so the blueprint stays portable across every PipelineKit-supported destination. (Sprint 7-G retrofits this pattern across all blueprints and into the BlueprintProposer system prompt.)
