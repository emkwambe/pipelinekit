## Purpose

Define the configuration system for PipelineKit.

`pipelinekit.yaml` is the canonical project configuration file.
It is human-readable, version-controllable, and executable without AI.
The Pydantic schema is the authoritative validator.
The pipeline.yaml contract defines what fields are required at runtime.

---

## Governing Rules

- `pipelinekit.yaml` is the single source of configuration truth
- Configuration must be human-readable YAML
- Configuration must be validated before any pipeline operation
- Invalid configuration must produce structured error codes, not stack traces
- Configuration is loaded from cwd by default
- Configuration must be executable without AI assistance

---

## The Canonical pipelinekit.yaml

This is the exact structure generated by `pipelinekit init` and validated by `pipelinekit validate`.

All 8 sections are required in the file.
6 of those sections are required by `contracts/pipeline.yaml`:
`name` (under pipeline), `runtime`, `ingestion`, `transformation`, `contracts`, `diagnostics`

```yaml
# pipelinekit.yaml
# PipelineKit Configuration
# See: https://pipelinekit.dev/docs/configuration

pipeline:
  name: my-project
  version: "0.1.0"
  description: ""

runtime:
  environment: local

ingestion:
  source:
    type: postgres
    host: localhost
    port: 5432
    database: ""
  destination:
    type: snowflake
    account: ""
    database: ""

transformation:
  enabled: false
  project_dir: ./transform

contracts:
  enabled: true
  directory: ./contracts

quality:
  enabled: false
  checks_dir: ./quality

diagnostics:
  enabled: false
  provider: none

notifications:
  enabled: false
  channels: []
```

## Required Capabilities

### Phase 1 (this SPEC)

- Load `pipelinekit.yaml` from cwd
- Parse YAML into Pydantic model
- Validate all 8 sections exist
- Validate all 6 contract-required fields exist
- Return structured validation errors with PK error codes
- Write default `pipelinekit.yaml` via `pipelinekit init`

### Phase 2 extension (runtime-engineer adds)

- Validate source/destination connection strings
- Validate `transformation.project_dir` exists when enabled
- Validate `contracts.directory` exists when enabled
- Environment variable substitution (e.g. `${DB_PASSWORD}`)

---

## Inputs

- `pipelinekit.yaml` file in cwd (or path provided by caller)

## Outputs

- `PipelineConfig` Pydantic model instance (success)
- `list[ConfigurationError]` with PK error codes (failure)

## Pydantic Schema

```python
# src/pipelinekit/config/schema.py

from pydantic import BaseModel, Field
from typing import Optional

class PipelineSection(BaseModel):
    name: str
    version: str = "0.1.0"
    description: str = ""

class RuntimeSection(BaseModel):
    environment: str = "local"

class SourceConfig(BaseModel):
    type: str
    host: Optional[str] = None
    port: Optional[int] = None
    database: Optional[str] = None
    account: Optional[str] = None

class IngestionSection(BaseModel):
    source: SourceConfig
    destination: SourceConfig

class TransformationSection(BaseModel):
    enabled: bool = False
    project_dir: str = "./transform"

class ContractsSection(BaseModel):
    enabled: bool = True
    directory: str = "./contracts"

class QualitySection(BaseModel):
    enabled: bool = False
    checks_dir: str = "./quality"

class DiagnosticsSection(BaseModel):
    enabled: bool = False
    provider: str = "none"

class NotificationsSection(BaseModel):
    enabled: bool = False
    channels: list[str] = []

class PipelineConfig(BaseModel):
    pipeline: PipelineSection
    runtime: RuntimeSection
    ingestion: IngestionSection
    transformation: TransformationSection
    contracts: ContractsSection
    quality: QualitySection
    diagnostics: DiagnosticsSection
    notifications: NotificationsSection
```

## Error Codes

| Code | Condition |
|---|---|
| `PK-CONFIG-001` | `pipelinekit.yaml` fails Pydantic validation |
| `PK-CONFIG-002` | Required section missing from config |
| `PK-CONFIG-003` | `pipelinekit.yaml` not found in cwd |
| `PK-CONFIG-004` | `pipelinekit.yaml` is not valid YAML |

All errors must be raised as `ConfigurationError(code, message, context)`.
Never raise bare exceptions.
Never print stack traces to the user.

## File Structure

```
src/pipelinekit/
├── config/
│   ├── __init__.py
│   ├── schema.py       PipelineConfig Pydantic models
│   └── loader.py       load_config(), config_exists(), write_default_config()
└── core/
    ├── __init__.py
    └── errors.py       PipelineKitError base, ConfigurationError, etc.
```

`core/errors.py` is shared infrastructure — both cli-engineer and runtime-engineer use it.

## Error Class Pattern

```python
# src/pipelinekit/core/errors.py

class PipelineKitError(Exception):
    def __init__(self, code: str, message: str, context: dict = {}):
        self.code = code
        self.message = message
        self.context = context
        super().__init__(f"[{code}] {message}")

class ConfigurationError(PipelineKitError):
    pass

class StateError(PipelineKitError):
    pass

class RuntimeError(PipelineKitError):
    pass

class ContractError(PipelineKitError):
    pass
```

## Constraints

- Python 3.11+
- Pydantic v2 for all models
- PyYAML for YAML parsing
- No environment variable substitution in Phase 1
- No remote config loading
- No config merging or inheritance
- Config is always loaded from a single `pipelinekit.yaml` file

---

## AI Guardrails

- AI may not modify `pipelinekit.yaml` automatically
- AI may suggest configuration changes — human approves
- Config validation is purely deterministic — no AI involvement

---

## Acceptance Criteria

```
✓ load_config() returns PipelineConfig on valid YAML
✓ load_config() raises ConfigurationError(PK-CONFIG-003) when file not found
✓ load_config() raises ConfigurationError(PK-CONFIG-004) on invalid YAML syntax
✓ load_config() raises ConfigurationError(PK-CONFIG-001) on schema violation
✓ write_default_config() creates valid pipelinekit.yaml that passes load_config()
✓ config_exists() returns True when file present, False when absent
✓ poetry run pytest tests/config/ → all tests pass
```
